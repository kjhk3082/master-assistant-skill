# 체계적인 디버깅

<HARD-GATE>
수정을 추측하지 마십시오. 근본 원인을 이해하기 전에 코드를 변경하지 마십시오.
</HARD-GATE>

## 4단계 디버깅 프로세스

### 1단계: 증상 문서화
```
에러 메시지: [정확한 에러 텍스트]
발생 위치: [파일:줄번호]
재현 방법: [단계별 재현 방법]
예상 동작: [무엇이 일어나야 했는가]
실제 동작: [무엇이 실제로 일어났는가]
```

### 2단계: 가설 수립
최소 2가지 가능한 원인을 나열합니다:
- 가설 A: [원인과 근거]
- 가설 B: [원인과 근거]

### 3단계: 검증 (수정 전)
각 가설을 검증합니다:
- 로그를 읽고 스택 트레이스를 분석합니다.
- 관련 코드를 읽습니다 (추측하지 않음).
- 필요한 경우 임시 로그를 추가합니다.
- 근본 원인을 확인합니다.

### 4단계: 최소한의 수정
근본 원인이 확인된 후에만:
- 가장 작은 수정을 적용합니다.
- 수정 전후를 비교합니다.
- 수정이 다른 것을 깨뜨리지 않는지 확인합니다.
- `docs/wiki/gotchas.md`에 문서화합니다.

## 핵심 규칙
- **절대 금지**: 근본 원인 확인 전 수정
- **절대 금지**: 여러 수정을 동시에 적용
- **반드시**: 까다로운 버그는 gotchas.md에 문서화
